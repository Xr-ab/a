## Day 1 · 变量、输入输出

### 重点概念

| 知识点 | 是什么 | 什么时候用 |
|--------|--------|-----------|
| **变量** | 给数据起个名字存起来 | 任何需要"记住一个值"的时候 |
| `str` | 文本字符串 | 处理名字、消息、任何文字内容 |
| `int` | 整数 | 计数、年龄、编号 |
| `float` | 小数 | 价格、百分比、计算结果 |
| `bool` | 真/假，`True`/`False` | 做判断、开关、条件 |
| **f-string** | 把变量嵌入字符串，`f"姓名：{name}"` | 拼接输出内容时，比 `+` 拼接更直观 |
| **`input()`** | 从键盘读用户输入 | 交互式程序 |
| **类型转换** | `int("123")` → 123，`float("3.14")` → 3.14 | `input()` 返回的一定是字符串，做数学运算前必须先转 |

### 为什么需要类型转换

```python
# input() 不管用户输入什么，都当成字符串返回
age = input("年龄：")  # 用户输入 22，age = "22"（字符串，不是数字）
# age + 1  → 报错！字符串不能加数字
age = int(age)  # 转成整数后才能做运算
```

### 检验标准

能独立写出接收两个数字并算出和、乘积、平均值的程序。

## Day 2 · 条件判断 + 循环

### 重点概念

| 知识点 | 是什么 | 什么时候用 |
|--------|--------|-----------|
| **`if/elif/else`** | 条件判断：满足 A 就做 X，否则做 Y | 需要根据情况走不同分支 |
| **`while` 循环** | 重复执行一段代码，直到条件不成立 | **不知道**具体要循环多少次（比如一直猜直到猜对） |
| **`for` 循环** | 对一个集合里的每个元素做同样的事 | **知道**要遍历几个东西（比如打印 1~9） |
| **`range(n)`** | 生成 0 到 n-1 的整数序列 | for 循环里控制次数 |
| **`break`** | 立即终止整个循环 | 找到目标了，不用继续了 |
| **`continue`** | 跳过当前这一轮，进入下一轮 | 这一轮不符合条件，但不影响后面的循环 |

### while vs for 怎么选

```
"循环几次？"
 ├─ 确定的次数 → for + range()
 └─ 不确定，等某个条件满足 → while
```

### 常见坑

- `input()` 返回字符串，跟数字比较前必须 `int()` 转
- `while True` 必须有 `break`，否则无限循环

### 检验标准

不查资料写出猜数字游戏（1-100，10次机会），能说清楚 `break` 和 `continue` 的区别。

## Day 3 · 函数 + 列表字典

### 重点概念

| 知识点 | 是什么 | 什么时候用 | 难易 |
|--------|--------|-----------|------|
| **函数 `def`** | 把一段代码打包，起个名字，可以反复调用 | 同样的逻辑要用多次，避免复制粘贴 | ⭐ |
| **参数** | 函数接收的输入值 | 让函数每次调用可以处理不同数据 | ⭐ |
| **`return`** | 函数把计算结果返还给调用方 | 需要拿到函数的执行结果 | ⭐ |
| **列表 `list`** | 有序的集合，`[元素1, 元素2, ...]` | 存多个同类数据（学生名单、成绩列表） | ⭐ |
| **字典 `dict`** | 键值对集合，`{"key": value}` | 存有结构的数据（一个人的姓名+成绩+年龄） | ⭐ |
| **列表推导式** | 一行代码生成新列表，`[x for x in list if 条件]` | 对列表做过滤/转换，比写 for 循环简洁 | ⭐⭐ |
| **生成器表达式** | 像列表推导式但逐个产出值，不一次性生成全部，`(x for x in ...)` | 数据量大时省内存，或直接喂给 `sum()`/`max()` | ⭐⭐⭐ |
| **`lambda`** | 没名字的迷你函数，`lambda 参数: 表达式` | 临时需要一个简单函数的地方，比如 `max(list, key=...)` | ⭐⭐⭐ |
| **`enumerate()`** | 遍历的同时自动编号 | 打印排行榜、序号列表 | ⭐⭐ |

### 函数为什么重要

```python
# 没有函数：同一件事写三遍
print("张三的成绩已添加")
print("李四的成绩已添加")
print("王五的成绩已添加")

# 有函数：写一遍，调三次
def notify(name):
    print(f"{name}的成绩已添加")
notify("张三")
notify("李四")
notify("王五")
```

### lambda 为什么需要

有些地方要求传一个"比较规则"，但不值得专门写个函数：

```python
# 不用 lambda：写一个完整函数
def by_score(s):
    return s["score"]
best = max(students, key=by_score)

# 用 lambda：一行搞定
best = max(students, key=lambda s: s["score"])
```

### 字典 vs 列表什么时候用

```
多个学生名 → 列表：["张三", "李四", "王五"]
一个学生的多种信息 → 字典：{"name": "张三", "score": 85, "age": 20}
多个学生+每人多种信息 → 列表里套字典：[{"name": "张三", ...}, {"name": "李四", ...}]
```

### 检验标准

不查资料写出学生成绩管理系统的四个函数（添加/平均分/最高分/展示）。

## Day 4 · 文件读写 + JSON

### 重点概念

| 知识点 | 是什么 | 什么时候用 |
|--------|--------|-----------|
| **`open("文件", "模式")`** | 打开一个文件准备读或写 | 需要读写本地文件 |
| **`"r"` 模式** | 只读 | 读取已有文件内容 |
| **`"w"` 模式** | 写入（会覆盖原内容） | 创建新文件或覆盖旧内容 |
| **`"a"` 模式** | 追加（不覆盖，写到末尾） | 往已有文件后面加内容 |
| **`with open()`** | 自动关闭文件的写法 | **永远用这个**，不用手动 `.close()` |
| **`encoding="utf-8"`** | 指定编码方式 | 有中文必须加，否则乱码 |
| **`json.dump(数据, 文件)`** | 把 Python 对象写入 JSON 文件 | 保存数据到硬盘 |
| **`json.load(文件)`** | 从 JSON 文件读取为 Python 对象 | 从硬盘恢复数据 |
| **`ensure_ascii=False`** | JSON 里正常显示中文 | 有中文时必加，否则变成 `\uXXXX` |
| **`indent=2`** | JSON 排版缩进 | 让文件人类可读，不写会挤成一行 |

### 为什么需要文件存储

```
程序运行时的数据在内存 → 程序关闭 → 内存清空 → 数据没了
                              ↓
                      存到硬盘上（文件） → 下次打开程序还能读回来
```

### 为什么用 with open 而不是 open + close

```python
# 手动管理：容易忘 close，文件可能损坏
f = open("test.txt", "r")
data = f.read()
f.close()  # ← 容易忘，程序如果在中间崩溃就更不会执行了

# with 写法：缩进结束自动关闭，即使中间出错也会关
with open("test.txt", "r", encoding="utf-8") as f:
    data = f.read()
# 缩进出去了，文件已自动关闭
```

### 三个必记的坑

1. **读文件**：`f.read()` 才是读内容 — 不是 `content = "文件名"`
2. **JSON 读**：`json.load(f)` 括号里要传文件对象 — 不是 `json.load()` 空着
3. **字典 key**：`dict["name"]` 要加引号 — `dict[name]` 是去找一个叫 name 的变量

### 检验标准

能把 Day 3 的成绩管理系统加上"启动时从 JSON 读，添加后自动写回"，程序关了再开数据还在。

## Day 5 · 异常处理 + 类与对象

### 异常处理

| 知识点 | 是什么 | 什么时候用 |
|--------|--------|-----------|
| **`try/except`** | 尝试执行，出错时有备用方案 | 知道某段代码可能出错，但不希望程序崩溃 |
| **`raise`** | 主动抛出错误 | 数据不合法时主动拒绝，让调用方知道"你给我的是错的" |
| **`finally`** | 不管成不成功，最后都执行 | 释放资源（如关闭数据库连接） |
| **`else`** | try 成功后执行 | try 成功才做的事，和 except 互斥 |

### try/except vs raise 的区别

```
try/except → 接住别人的错误（防守）
raise      → 主动报错，告诉别人"你错了"（进攻）
```

```python
# 场景：用户输入成绩
# 防守：输入的可能是"abc"，不是数字
try:
    score = int(input("成绩："))
except ValueError:
    print("请输入数字！")

# 进攻：输入的是数字 200，但成绩不能超过 100
if score > 100:
    raise ValueError("成绩必须在0-100之间")  # 主动拒绝
```

### 类与对象

| 知识点 | 是什么 | 大白话 |
|--------|--------|--------|
| **`class`** | 把相关数据和方法打包成一体 | 蓝图/模板 |
| **对象（实例）** | 根据 class 创建出来的具体个体 | 按蓝图造出来的实物 |
| **`__init__()`** | 创建对象时自动执行，做初始化 | 出厂设置，创建对象时自动跑 |
| **`self`** | 方法里代表"当前这个对象自己" | "我自己"的代词 |
| **实例方法** | class 里定义的函数，第一个参数必须是 `self` | 这个对象能干的事 |
| **`_method()`** | 前下划线 = 约定为内部使用 | "别在外面调我，我是内部干活的" |

### 为什么需要类：一个具体例子

```python
# 没有类：数据和功能是散的，同时管两个班就乱了
students_a = []
students_b = []
def add_a(name): students_a.append(name)
def add_b(name): students_b.append(name)
# ↑ 每多一个班，函数都要写一套，很容易混

# 用类：每个班是一个独立的对象，各自管理自己的数据
class ClassManager:
    def __init__(self):
        self.students = []
    def add(self, name):
        self.students.append(name)

一班 = ClassManager()
二班 = ClassManager()
一班.add("张三")  # 只影响一班，不影响二班
二班.add("李四")  # 只影响二班
```

### 函数 vs 类怎么选

```
你的程序需要同时管理多个同类实体吗？

不需要（只处理一份数据） → 用函数就够了
需要（比如多个班级、多个用户、多个文件管理器）→ 用类
```

### 两个常见坑

1. **`self` 忘写** — 方法里所有访问自身数据的操作必须 `self.xxx`，漏了 `self.` 就找不到变量
2. **`__init__` 是双下划线** — `__init__` 不是 `_init_`，前后各两个下划线，这是 Python 的"特殊方法"标记

### 检验标准

能把 Day 4 的成绩管理系统用 class 重写，加上 try/except 防护（非法分数拒绝，删除不存在的人报错）。

## Day 6 · 模块导入 + 虚拟环境

### 重点概念

| 知识点 | 是什么 | 大白话 |
|--------|--------|--------|
| **`import`** | 把另一个 Python 文件里的代码"借"过来用 | 工具箱里拿工具，不用重新造 |
| **`from X import Y`** | 只借用某个文件里的某个具体东西 | 只拿扳手，不搬整个工具箱 |
| **`if __name__ == "__main__"`** | 判断当前文件是不是程序的入口 | "如果是自己运行就执行，如果被别人 import 就别乱动" |
| **虚拟环境 `venv`** | 每个项目独立的 Python 运行环境 | 每个项目有自己的 pip 包，互不污染 |
| **`pip install`** | 安装第三方包 | 下载别人写好的工具放到自己的项目 |
| **`requirements.txt`** | 记录项目依赖哪些包 | 换电脑后一条命令就能装回所有包 |

### 为什么拆文件：一个核心对比

```python
# 错误理解 —— import 后面又重写一遍类（等于抄了一遍，白 import 了）
from student_manager import StudentManager

class StudentManager:          # ← 多余！import 已经拿到这个类了
    def __init__(self): ...

# 正确写法 —— import 后直接用它，不用再定义
from student_manager import StudentManager

manager = StudentManager()     # 直接用
manager.add("张三", 85)        # 直接调
print(manager.average())       # 直接算
```

### `if __name__ == "__main__"` 是干什么的

```python
# 一个文件被 import 时，Python 会执行它里面所有的顶层代码
# 如果不加 if 保护，被 import 时就跳出来一堆 print/测试

# student_manager.py 底部加这个：
if __name__ == "__main__":
    # 里面的代码只有"直接运行 student_manager.py"才执行
    # 被别的文件 import 时不会执行
    print("这是模块自测代码，被 import 时不会看到我")
```

**规则**：所有被 import 的文件，要在文件底部写 `if __name__ == "__main__"`，把测试代码包起来。

### 为什么需要虚拟环境

```
没有虚拟环境：
  项目A 需要 pandas 1.0 → pip install
  项目B 需要 pandas 2.0 → 冲突！只有一个能装

有虚拟环境：
  d:/项目A/venv/ → pandas 1.0
  d:/项目B/venv/ → pandas 2.0
  各用各的，互不干扰
```

### 必须记住的命令

```bash
# 创建虚拟环境
python -m venv venv

# 激活（Windows）
venv\Scripts\activate

# 安装包
pip install requests python-dotenv

# 导出依赖清单
pip freeze > requirements.txt

# 从清单恢复依赖（别人拿到你的代码时）
pip install -r requirements.txt
```

### 三个常见坑

1. **`import` 后不要重写** — `from X import StudentManager` 就是拿到了，不需要再 `class StudentManager`
2. **`.gitignore` 里必须有 `venv/`** — 虚拟环境文件夹巨大，绝对不能提交到 Git
3. **`import` 的冒号问题** — `from student_manager import StudentManager` 末尾不要加冒号，它不是函数/类/if 的开头

### 检验标准

能创建两个独立文件，`main.py` 只写调用逻辑不重复定义类，导入不报错，程序能跑通。
